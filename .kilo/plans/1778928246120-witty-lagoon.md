# Plan: Fix System Update JSON Error & Clear Database Failures

## Bug 1 ŌĆö "Unexpected end of JSON input" on system update POST

**File:** `app/update-sistem/UpdateSistemClient.tsx`

`handleSubmit` (line 55ŌĆō61) calls `await res.json()` on the `/api/system-update` POST response with **no try/catch**. If the server returns a non-JSON/empty body (e.g. server error, framework hiccup, unexpected redirect), the promise rejects with `SyntaxError: Unexpected end of JSON input`, the error is unhandled, and the user sees a generic crash instead of a meaningful message.

The same issue exists in `fetchHistory` (line 33): `await res.json()` is bare and can throw on a non-JSON response.

## Bug 2 ŌĆö "Clear database gagal" for Administrator

Two compounding issues:

**Issue 2a ŌĆö silent error swallowing in client**  
`ExportClient.tsx` line 104:
```ts
const json = await res.json().catch(() => ({}))
```
When `res.json()` throws (empty body, HTML redirect, non-JSON content-type), the `.catch(() => ({}))` swallows it and returns `{}`. `json.error` is `undefined`, so `toast.error(json.error || 'Clear database gagal')` always shows the **generic** error with no diagnostic info. The root cause (server returning empty/HTML body) is entirely hidden from the user.

The fix: do **not** swallow the parse error. Catch it properly and include the error type/message in the toast so the user and developer knows **why** the operation failed.

**Issue 2b ŌĆö middleware role comparison mismatch risk**  
`middleware.ts` line 43:
```ts
if (pathname.startsWith('/admin') && session.role !== 'administrator') {
```
This is an **un-trimmed strict equality** check against `session.role` from the JWT. If the stored role ever contains leading/trailing whitespace (e.g. `' administrator'` from a malformed seed/import), the middleware check fires incorrectly and **redirects** the POST to `/dashboard` (an HTML redirect response). The client then tries `await res.json()` on HTML ŌĆö producing the `SyntaxError: Unexpected end of JSON input` crash. This is identical to Bug 1's mechanism.

Similarly, `middleware.ts` line 35 (`/log`) and `app/api/log/route.ts` line 6 are also un-trimmed strict comparisons against `x-user-role`.

## Changes

### Change 1 ŌĆö `app/update-sistem/UpdateSistemClient.tsx`
- Wrap `handleSubmit`'s `await res.json()` in try/catch; on error toast the actual message before checking `res.ok`
- Add `.catch(() => null)` on `fetchHistory`'s `await res.json()` to prevent crash on non-JSON responses

### Change 2 ŌĆö `app/export/ExportClient.tsx` ŌĆö `handleClearDatabase`
- Remove the silencing `.catch(() => ({}))` on `await res.json()`
- After catching a JSON parse error, extract `res.status` and `res.statusText` and build a meaningful diagnostic toast (e.g. "Server error 500 ŌĆö cek log server")
- Add `console.error(...)` logging so the real server error is visible in browser devtools

### Change 3 ŌĆö `middleware.ts`
- Replace all three raw `!== 'administrator'` checks with `?.trim() !== 'administrator'` so a role string with accidental whitespace still passes the admin check correctly:
  - Line 35: `/log` role check
  - Line 38: `/api/log` role check
  - Line 43: `/admin` role check

### Change 4 ŌĆö `app/api/admin/clear-database/route.ts`
- Add `try/catch` wrapping around the entire handler body so any unhandled runtime exception (DB disconnect, write permission error, etc.) produces a proper **JSON 500 response** instead of an empty/HTML body

### Change 5 ŌĆö `app/api/admin/clear-database/route.ts` ŌĆö `getCtx`
- Change `userRole: req.headers.get('x-user-role') || ''` to `userRole: (req.headers.get('x-user-role') || '').trim()` ŌĆö trims the role before passing it to `isAdministrator`, and leave `isAdministrator` as-is (it uses strict equality, which is correct once the input is normalised)
